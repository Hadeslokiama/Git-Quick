Configuration

Essential Global Configuration

These Git settings are essential because they tell Git who you are, so your commits include your name and email. This ensures your work is properly attributed, linked to your GitHub account, and consistent across all your projects.

Git Bash / Terminal / PowerShell

# Enter command line in Git Bash, Terminal, or Powershell

git config --global user.name "Your Name"

git config --global user.email "you@example.com"

Security

Step-by-Step SSH Setup

An SSH key lets your computer securely connect to GitHub without using a password. It provides safe, encrypted authentication, makes Git operations like push/pull easier, and is more secure than using a password.

terminal — zsh

# Generate an SSH key

ssh-keygen -t ed25519 -C "you@example.com"

 

# Once you run the command press Enter to accept the default file location.

# Next, you'll be prompted to enter a passphrase. You can choose to set one

# for added security or leave it empty for no passphrase (by pressing Enter twice).

 

# Start the ssh-agent in the background

eval "$(ssh-agent -s)"

 

# Add your SSH key to the agent

ssh-add ~/.ssh/id_ed25519

 

# Check if the key was added successfully

ssh-add -l

 

# Copy the SSH key to your clipboard

cat ~/.ssh/id_ed25519.pub

# (Copy the full output starting with "ssh-ed25519 ...")

 

# Add your Generated SSH key to GitHub

# Go to GitHub → Settings → SSH and GPG keys → New SSH key

# Give it a title, paste the public key, then click "Add SSH key".

1

Generate SSH Key

Open your terminal and run the following command to create a new Ed25519 key. Press Enter to accept the default file location, then optionally set a passphrase.

ssh-keygen -t ed25519 -C "your_email@example.com"
2

Add to SSH Agent

Ensure the agent is running and add your private key to the local manager. Then verify the key was added successfully.

eval "$(ssh-agent -s)" ssh-add ~/.ssh/id_ed25519 ssh-add -l
3

Add to GitHub

Copy your public key and paste it into your GitHub SSH settings. Copy the full output starting with "ssh-ed25519 ...".

cat ~/.ssh/id_ed25519.pub

open_in_new Then follow these steps on GitHub:

  • 1 Go to GitHub → Settings → SSH and GPG keys
  • 2 Click New SSH key
  • 3 Give it a title and paste the public key
  • 4 Click Add SSH key to save

Git Cheat Sheet

Your essential daily command reference.

Initialize Repository

The git init command initializes a new Git repository on your local machine. It creates a new .git subdirectory in your project folder, which contains all the necessary metadata and object database for your version control.

git init

Initialized empty Git repository in /Users/me/myproject/.git/

Output: Initialized empty Git repository
cloud_download

Clone Repository

The git clone command copies a remote repository to your local machine. The URL can be found on GitHub by clicking the "Code" button. You may choose either HTTPS or SSH based on your authentication preference.

# Via HTTPS

git clone https://github.com/user/repo.git

# Via SSH

git clone git@github.com:user/repo.git

Cloning into 'repo'...

remote: Enumerating objects: 42, done.

remote: Counting objects: 100% (42/42), done.

Receiving objects: 100% (42/42), 12.34 KiB | 2.00 MiB/s, done.

Output: Cloning into 'repo'...
info

Check Repository Status

The git status command displays the state of the working directory and staging area. It lets you see which changes have been staged, which haven't, and which files aren't being tracked by Git.

git status

On branch main

Your branch is up to date with 'origin/main'.

# If file is not added to staging area.

Changes not staged for commit:

  modified: index.html

# If file is added to staging area.

Changes to be committed:

  modified: index.html

Output: On branch main
add_circle

Add Files to Staging Area

The git add command stages changes so they are ready to be committed. You can add individual files or everything at once.

git add index.html # Adds only the specific file

git add . # Adds all changes

$ git add index.html

$ git add .

Changes staged for commit:

  new file: index.html

  modified: styles.css

Output: Changes staged for commit
add_task

Commit Changes

The git commit command permanently saves staged changes with a message. It creates a new commit that includes all staged changes and helps keep a history for collaboration using the global commit convention.

git commit -m "feat(auth): add Google login button" -m "Integrates OAuth2 so users can log in using their Google account. Refs: #24"

[main 3fae123] Update layout and styles

3 files changed, 52 insertions(+), 10 deletions(-)

Output: [main ab12] message...
edit

Amend Last Commit

The git commit --amend command updates the most recent commit by adding new changes or fixing the commit message. Avoid amending commits that have already been pushed as it rewrites history.

git commit --amend -m "Fix: correct typo in layout"

[main 4bd2c18] Fix: correct typo in layout

Date: Tue Feb 4 12:22:10 2025

Output: [main 4bd2c18] amended commit
call_split

Create and Switch Branches

Branches allow you to work on new features without affecting the main codebase. Use git branch and git checkout to keep work organized and isolated from stable code.

git branch feature-login

git checkout feature-login

# or shorthand

git checkout -b feature-login

Switched to a new branch 'feature-login'

# Verify current branch

* feature-login

  main

Output: Switched to new branch 'feature-login'
cloud_sync

Fetch Remote Changes

The git fetch command downloads new commits, branches, and tags from a remote repository without modifying your current working directory or local branches. It updates your remote-tracking references (e.g., origin/main or upstream/main) so you can review changes before merging or rebasing.

git fetch origin

# Example with upstream remote

git fetch upstream

remote: Counting objects: 100% (5/5), done.

Unpacking objects: 100% (3/3), done.

From https://github.com/original-owner/project

a1b2c3d..e4f5g6h main -> upstream/main

Output: remote: Enumerating objects...
merge

Merge Branches

The git merge command combines changes from one branch into another. It can create a merge commit or fast-forward, depending on the branch history. Conflicts may appear if both branches modify the same parts of a file.

git checkout main

git merge branch-name

Updating 1a2b3c..4d5e6f

Fast-forward

login.html | 20 ++++++++++++++++++++

Output: Updating a1b2...c3d4
rebase

Rebase Branches

The git rebase command rewrites history by moving your branch onto the latest version of another branch. It creates a cleaner, linear commit history. Avoid rebasing public or shared branches to prevent conflicts for others.

git checkout branch-name

git rebase main

Successfully rebased and updated refs/heads/branch-name.

# Rebase progress

Rebasing (1/3)

Rebasing (2/3)

Rebasing (3/3) done.

Output: Successfully rebased onto main
upload

Push Changes to Remote

The git push command uploads your commits to a remote repository like GitHub. It makes your changes available to others and updates the remote branch. If your branch is behind the remote, Git may require a pull first.

git push origin main

Pull Latest Changes

The git pull command fetches and merges changes from a remote repository into your current branch. It keeps your local work updated with the team's changes. Conflicts may occur if your local changes conflict with the remote version.

git pull origin main

Stash Changes

The git stash command temporarily saves uncommitted changes so you can work on something else. It lets you switch branches or pull updates safely. You can reapply stashed changes later using git stash apply.

git stash

git stash apply

Saved working directory and index state

WIP on main: 3fae123 feat: add login page

# After git stash apply

On branch main

Changes not staged for commit:

  modified: login.html

Output: Saved working directory state WIP on main
history

The git log command shows the full commit history with details. It helps track changes and understand the project's evolution. You can use extra options to visualize or simplify the history.

git log

Standards

Global Commit Convention

Global Commit Convention (often called Commit Message Convention or Commit Guidelines) is a standardized way of writing Git commit messages. It's important because it ensures your commit history is clear, consistent, and easy to understand, especially in collaborative projects.

label Common Commit Types

Type Title Description
feat Feature Used when adding a new feature to the project.
fix Bug Fix Used when fixing a bug or issue in the code.
docs Documentation Used for documentation updates only.
style Styling Formatting changes that do not affect code logic (spaces, indentation, semicolons).
refactor Refactor Code restructuring without adding features or fixing bugs.
perf Performance Used for performance improvements.
test Tests Adding or updating tests only.
build Build Changes to build tools or dependencies.
ci CI Changes related to continuous integration pipelines (GitHub Actions, Jenkins, etc.).
chore Chore Routine tasks or maintenance that don't affect src or test files.
revert Revert When reverting a previous commit.

rule Recommended Rules

commit-format

type(scope): short summary (<=50 chars)

 

A more detailed description of the change, explaining

why it was made and any additional context.

Wrap lines at ~72 characters.

 

Refs: #123

code Example

# A backslash \ at the end of a line tells the shell the command continues on the next line for Linux/MacOS only.

git commit -m "feat(auth): add Google login button" -m "A more detailed description of the change, explaining why it was made and any additional context. This change integrates OAuth2 authentication so users can log in using their Google account. Refs: #24"
Community

How to Contribute

This section explains the standard workflow for contributing to a repository using a fork-and-pull request model. It follows best practices used in collaborative projects.

01

Fork the Repository

Forking creates your own copy of the repository under your GitHub account. This allows you to make changes without affecting the original project.

  • arrow_right Go to the repository on GitHub.
  • arrow_right Click the Fork button.
  • arrow_right Wait for GitHub to create your copy.
02

Clone Your Fork

Clone your fork to your local machine.

git clone https://github.com/your-username/project.git

cd project

Add the original repository as upstream to stay updated.

git remote add upstream https://github.com/original-owner/project.git

03

Create a New Branch

Never work directly on main. Create a separate branch for your feature or fix.

git checkout -b branch-name

04

Sync with Latest Changes

Keep your branch updated before making changes.

git fetch upstream

git checkout main

git merge upstream/main

git checkout branch-name

git rebase main

05

Make Your Changes

Modify files as needed. Always check status before committing.

git status

06

Stage Changes

Add files to the staging area.

git add .

# or add specific file

git add filename.ext

07

Commit Changes

Create a commit following the global commit convention.

git commit -m "feat(scope): short summary"

08

Push to Your Fork

Upload your branch to GitHub.

git push origin branch-name

09

Open a Pull Request

Open a Pull Request (PR) on GitHub to propose your changes.

  • arrow_right Go to your fork.
  • arrow_right Click Compare & Pull Request.
  • arrow_right Ensure base repository is the original project.
  • arrow_right Write a clear title and description.
