Git Setup & Cheat Sheet

Quick guide to set up Git and a compact cheat sheet including a global commit convention.

Install Git

  • Windows: Download from git-scm.com and run the installer.
  • macOS: Install via Homebrew: brew install git or download from git-scm.com.
  • Linux: Use your package manager, e.g. sudo apt install git or sudo dnf install git.

Essential Global Configuration

These Git settings are essential because they tell Git who you are, so your commits include your name and email. This ensures your work is properly attributed, linked to your GitHub account, and consistent across all your projects.

# Enter command line in Git Bash, Terminal, or Powershell
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

Step-by-Step SSH Setup

An SSH key lets your computer securely connect to GitHub without using a password. It provides safe, encrypted authentication, makes Git operations like push/pull easier, and is more secure than using a password.

# Generate an SSH key
ssh-keygen -t ed25519 -C "you@example.com"

# Once you run the command press Enter to accept the default file location.
# Next, you'll be prompted to enter a passphrase. You can choose to set one for added security or leave it empty for no passphrase (by pressing Enter twice).

# Start the ssh-agent in the background
eval "$(ssh-agent -s)"

# Add your SSH key to the agent
ssh-add ~/.ssh/id_ed25519

# Check if the key was added successfully
ssh-add -l

# Copy the SSH key to your clipboard
cat ~/.ssh/id_ed25519.pub
# (Copy the full output starting with "ssh-ed25519 ...")

# Add your Generated SSH key to GitHub
# Go to GitHub → Settings → SSH and GPG keys → New SSH key
# Give it a title, paste the public key, then click "Add SSH key".

Git Cheat Sheet

The git init command initializes a new Git repository on your local machine. It creates a new .git subdirectory in your project folder, which contains all the necessary metadata and object database for your version control.
Command:
git init
Output:
Initialized empty Git repository in /Users/me/myproject/.git/

The git clone command copies a remote repository (e.g., from GitHub) to your local machine. The URL specifies the location of the repository you want to clone. Which can be found on the repository page on GitHub by clicking the "Code" button and copying the URL provided. You may choose either the HTTPS or SSH URL based on your authentication preference.
Command:
# Via HTTPS 
git clone https://github.com/user/repo.git
# Via SSH 
git clone git@github.com:user/repo.git
Output:
Cloning into 'repo'...
remote: Enumerating objects: 42, done.
remote: Counting objects: 100% (42/42), done.
remote: Compressing objects: 100% (32/32), done.
Receiving objects: 100% (42/42), 12.34 KiB | 2.00 MiB/s, done.

The git status command displays the state of the working directory and the staging area. It lets you see which changes have been staged, which haven’t, and which files aren’t being tracked by Git. This is useful for understanding what will be included in the next commit.
Command:
git status
Output:
On branch main
Your branch is up to date with 'origin/main'.

# If file is not added to staging area.
Changes not staged for commit:
  (use "git add ..." to update what will be committed)
  (use "git restore ..." to discard changes in working directory)
        modified:   index.html
no changes added to commit (use "git add" and/or "git commit -a")

# If file is added to staging area.
Changes to be committed:
  (use "git restore --staged ..." to unstage)
        modified:   index.html

The git add command stages changes so they are ready to be committed. You can add individual files or everything at once.
Command:
git add index.html # Adds only the specific file to staging area.
git add . # Adds all changes to staging area.

The git commit command permanently saves staged changes in the repository with a message. It creates a new commit that includes all changes added to the staging area. Creating a commit helps you keep a history of changes and makes it easier to collaborate with others by using the global commit convention.
Command:
git commit -m "feat(auth): add Google login button" -m "A more detailed description of the change, explaining why it was made and any additional context. This change integrates OAuth2 authentication so users can log in using their Google account. Refs: #24"
Output:
[main 3fae123] Update layout and styles
 3 files changed, 52 insertions(+), 10 deletions(-)

The git commit --amend command updates the most recent commit by adding new changes or fixing the commit message. It replaces the previous commit with a new one instead of creating a separate commit. Avoid amending commits that have already been pushed because it rewrites history.
Command:
git commit --amend -m "Fix: correct typo in layout"
Output:
[main 4bd2c18] Fix: correct typo in layout
 Date: Tue Feb 4 12:22:10 2025

Branches allow you to work on new features without affecting the main codebase. You can create and switch branches with git branch and git checkout. They help keep work organized and isolated from the stable code.
Command:
git branch feature-login
git checkout feature-login
# or
git checkout -b feature-login

The git merge command combines changes from one branch into another. It can create a merge commit or fast-forward, depending on the branch history. Conflicts may appear if both branches modify the same parts of a file.
Command:
git checkout main
git merge feature-login
Output:
Updating 1a2b3c..4d5e6f
Fast-forward
 login.html | 20 ++++++++++++++++++++

The git rebase command rewrites history by moving your branch onto the latest version of another branch. It creates a cleaner, linear commit history. Avoid rebasing public/shared branches to prevent conflicts for others.
Command:
git checkout feature-login
git rebase main

The git pull command fetches and merges changes from a remote repository into your current branch. It keeps your local work updated with the team’s changes. Conflicts may occur if your local changes conflict with the remote version.
Command:
git pull origin main

The git push command uploads your commits to a remote repository (like GitHub). It makes your changes available to others and updates the remote branch.If your branch is behind the remote, Git may require a pull first.
Command:
git push origin main
Output:
Enumerating objects: 5, done.
Counting objects: 100% (5/5), done.
Delta compression using up to 4 threads
Compressing objects: 100% (3/3), done.
Writing objects: 100% (3/3), 544 bytes | 544.00 KiB/s, done.
Total 3 (delta 1), reused 0 (delta 0), pack-reused 0 (from 0)
remote: Resolving deltas: 100% (1/1), completed with 1 local object.
To https://github.com/YourProject/Project.git
   d327d40..326e145  main -> main

The git stash command temporarily saves uncommitted changes so you can work on something else. It lets you switch branches or pull updates safely. You can reapply stashed changes later using git stash apply.
Command:
git stash
git stash apply

The git log command shows the full commit history with details. It helps track changes and understand the project’s evolution. You can use extra options to visualize or simplify the history.
Command:

git log

Global Commit Convention

Global Commit Convention (often called Commit Message Convention or Commit Guidelines) is a standardized way of writing Git commit messages. It’s important because it ensures your commit history is clear, consistent, and easy to understand, especially in collaborative projects.

Common Commit Types

  • feat – Used when adding a new feature to the project.
  • fix – Used when fixing a bug or issue in the code.
  • docs – Used for documentation updates only.
  • style – Formatting changes that do not affect code logic (spaces, indentation, semicolons).
  • refactor – Code restructuring without adding features or fixing bugs.
  • perf – Used for performance improvements.
  • test – Adding or updating tests only.
  • build – Changes to build tools or dependencies.
  • ci – Changes related to continuous integration pipelines (GitHub Actions, Jenkins, etc.).
  • chore – Routine tasks or maintenance that don’t affect src or test files.
  • revert – When reverting a previous commit.

Recommended rules

type(scope): short summary (<=50 chars) 
        
A more detailed description of the change, explaining why it was made and any additional context. Wrap lines at ~72 characters. 

Refs: #123

Example

# A backslash \ at the end of a line tells the shell the command continues on the next line for Linux/MacOS only.

git commit -m "feat(auth): add Google login button" -m "A more detailed description of the change, explaining why it was made and any additional context. This change integrates OAuth2 authentication so users can log in using their Google account. Refs: #24"
